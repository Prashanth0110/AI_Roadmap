GitHub Copilot

From autocomplete to full agentic coding partner

How Copilot Works in 2026

Copilot has three modes in VS Code:

Ask Mode

Chat

Q&A about your code. Doesn't edit files directly.

Edit Mode

Edits

Multi-file edits on files you specify. You control scope.

Agent Mode ★

Agent

Autonomous: plans tasks, runs terminal, iterates, self-heals errors across your whole codebase.

Configuration: copilot-instructions.md

Copilot reads a project-wide instruction file at .github/copilot-instructions.md. Since July 2025 it also supports scoped files:

.github/instructions/typescript.instructions.md
--- applyTo: "**/*.tsx" ---
Use functional components with TypeScript interfaces.
Always use server components unless client interactivity is needed.
Import from @/components — never use relative paths.

--- applyTo: "**/*.java" ---
Use Spring Boot 3.x patterns.
Always add @Slf4j for logging. Never use System.out.println.
Use record types for immutable DTOs.

Confidence & Assumption Control

⚠️ The Core Problem Copilot assumes when it doesn't know. It will invent import paths, function signatures, and library versions. These "confident hallucinations" are the #1 source of bugs.

Add these to your copilot-instructions.md to reduce assumptions:

.github/copilot-instructions.md — Confidence Rules
## Assumption Rules
- If you are not certain about an import path, write a TODO comment instead of guessing.
- Never invent method signatures — only use what exists in the codebase.
- If a library version is unknown, use the version in package.json / pom.xml.
- Prefer explicit over clever. Never sacrifice readability for brevity.
- When asked to refactor, preserve all existing behavior unless told otherwise.
- Ask clarifying questions rather than assume intent on ambiguous prompts.

Agent Mode Settings (VS Code)

1

Open Agent Mode

Press Ctrl+Alt+I → click the mode dropdown at top → select Agent

2

Choose Your Model

Agent mode supports Claude Sonnet 4.6, GPT-4.1, and Gemini 2.5 Pro depending on your plan. Pick from the model dropdown next to the mode selector.

3

Enable MCP Servers

In Settings → GitHub Copilot → MCP Servers, add local or remote MCP endpoints. The GitHub local MCP server lets Copilot search issues and create PRs directly from VS Code.

4

Auto-Approve (JetBrains)

Settings → GitHub Copilot → Chat → Auto Approve. Controls whether the agent can run terminal commands without pausing for confirmation.

📌 Org Admins For Copilot Business/Enterprise: Settings → Copilot → Policies → enable "Copilot coding agent" and select which repos it applies to. Agent mode in the IDE is enabled by default but can be toggled per-org in the AI Controls tab.

Cloud Coding Agent (GitHub Issues → PRs)

Copilot's cloud agent (separate from IDE agent mode) can be assigned GitHub Issues and will open PRs autonomously. To enable GitHub Actions to run on its PRs without manual approval: Repo Settings → Code & Automation → Copilot → Cloud Agent → toggle workflow auto-run.

🔴 Warning Enabling auto-run for GitHub Actions workflows means unreviewed AI code can access your repo secrets. Always inspect proposed .github/workflows/ changes before approving.

Claude Code

Anthropic's terminal-first agentic coding tool

Architecture Overview

Claude Code is a CLI + VS Code / JetBrains extension that runs Claude as a coding agent with full terminal access. It uses a multi-agent architecture:

General Agent

Main

Full tool access. Inherits your CLAUDE.md context. Orchestrates subagents.

Explore Agent

Read-Only

Fast codebase search. No file edits. Used for navigation and discovery.

Plan Agent

Architect

Software design and implementation planning. Produces structured task lists before coding.

Configuration File Hierarchy

Folder Structure
# Files loaded in merge order (global → project → subdirectory)
~/.claude/CLAUDE.md          # Your personal global rules (all projects)
./CLAUDE.md                  # Project root (shared via git)
./src/CLAUDE.md              # Subdirectory-scoped rules

.claude/
  settings.json              # Tool permissions, hooks config
  rules/
    java.md                  # Path-scoped rules
    tests.md
  agents/
    code-reviewer.md         # Custom subagent definitions
  skills/
    my-feature/SKILL.md      # Reusable skills with slash commands

What Goes in CLAUDE.md

⚠️ Instruction Budget Claude Code's built-in system prompt uses ~50 instructions. You have ~100–150 left. When CLAUDE.md grows too large, instructions get lost — not from token overflow, but from attention degradation. Keep it focused.
CLAUDE.md — Example
# Project: AirlineOps Microservices

## Tech Stack
Java 17, Spring Boot 3.2, Kafka, Solace PubSub+, DynamoDB, AWS

## Coding Rules
- Always use @Slf4j — never System.out.println
- Never assume a method exists — read the file first
- Every new service class needs a JUnit 5 + Mockito test
- Never modify pom.xml versions without explicit instruction

## Assumption Rules
- If uncertain about a Solace topic hierarchy, ask instead of guessing
- Never invent DynamoDB attribute names — check schema docs first

@import .claude/rules/kafka.md
@import .claude/rules/aws.md

Subagent Permission Controls (settings.json)

.claude/settings.json
{
  "permissions": {
    "allow": ["Bash(git status)", "Bash(mvn test)"],
    "deny":  ["Bash(rm -rf)", "Bash(kubectl delete)"]
  },
  "hooks": {
    "PreToolUse": [{
      "matcher": "Edit",
      "hooks":   [{ "type": "command", "command": "echo 'File edit: $TOOL_INPUT'" }]
    }]
  }
}
💡 Subagent Safety Rule Restrict subagents to read-only tool sets: remove Edit, Write, and Bash from their frontmatter tools: list. Let the parent agent handle all file writes after reviewing subagent output.

Amazon Kiro

AWS's spec-driven agentic IDE — code is a build artifact

📌 Core Philosophy Difference Copilot and Claude Code optimize for speed of output — prompt → code fast. Kiro optimizes for traceability and structure — requirements → design → tasks → code. For production AWS systems, Kiro is the most deliberate tool available.

The Spec-Driven Workflow

1

Write a Feature Prompt

Natural language description: "Add real-time flight delay notifications for passenger alerts via SNS."

2

Kiro Generates a Spec

Kiro creates a formal requirements doc in EARS notation (Easy Approach to Requirements Syntax), a design doc, and a structured task list.

3

You Review & Approve

Before a single line of code is written, you sign off on what will be built. This is the key differentiator — the spec is source of truth, code is the artifact.

4

Agents Generate Code

Claude Sonnet (reasoning-heavy specs) + Amazon Nova (high-throughput code gen) via Bedrock execute the tasks. Tests are generated alongside code.

Steering Files (The "CLAUDE.md" of Kiro)

Steering files live at .kiro/steering/ and give Kiro persistent context across sessions and contributors:

.kiro/steering/engineering-standards.md
## Security
All database queries must use parameterized statements.
Never log PII — mask passenger names and emails in all log output.

## Observability
Always use the internal LoggingService — never console.log or System.out.println.
Add Dynatrace trace IDs to all Kafka message headers.

## AWS
All Lambda functions must have Dead Letter Queues configured.
Use us-east-1 as primary, us-west-2 as DR region for all new resources.

Agent Hooks (Event-Driven Automation)

Hooks fire on IDE events and run predefined agent actions in the background — before code reaches CI/CD:

On File Save

Auto-sync Docs

When you save a service class, Kiro automatically updates the corresponding spec/design doc to reflect the implementation.

On PR Open

Test Generation

When a PR is opened, Kiro generates missing unit tests and checks that code matches the approved spec.

On Commit

Compliance Check

Verifies steering rules: parameterized SQL, no PII logging, correct logging library.

On File Create

Boilerplate Injection

New Java class? Kiro adds the standard header, @Slf4j, copyright comment, and test stub automatically.

Confidence & Assumption Settings in Kiro

⚠️ Kiro's Secret Weapon The spec review step is your assumption control. By approving the requirements and design before code is written, you've already eliminated the biggest source of assumptions. Kiro can't build the wrong thing if you approved what it will build.
.kiro/steering/no-assumptions.md
## Assumption Policy
- If a requirement is ambiguous, generate a clarifying question in the spec — do NOT assume.
- Never introduce a new dependency without adding it to the spec's tech-choices section.
- When referencing an existing service, read its interface definition file before using it.
- If a DynamoDB table schema is not in the spec, halt and ask before writing data access code.

Agent vs Prompt .md

The most misunderstood distinction in agentic AI

📌 The Core Distinction A prompt is a one-shot instruction inserted into a conversation. An agent is an entity with its own context window, tool access, and goal-completion loop that runs until the task is done or it hits a boundary.

Prompt (.md as template)

  • Inserted into your conversation
  • Runs once — you see the output
  • No isolated context window
  • No tool access of its own
  • Example: /review slash command
  • Like a chef reading a recipe card
vs

Agent (.md as definition)

  • Spawned with its own context window
  • Loops until goal is achieved
  • Has its own tool set (read, write, bash)
  • Runs in parallel or in sequence
  • Example: Explore subagent in Claude Code
  • Like hiring a contractor with a brief

What's Actually in an Agent .md File

.claude/agents/kafka-reviewer.md — Agent Definition
---
name: kafka-reviewer
description: "Reviews Kafka consumer/producer code for schema contract compliance and DLQ configuration"
tools:
  - Read          # allowed
  - Bash(grep)    # allowed
  # Edit is NOT listed — this agent is read-only
---

You are a Kafka code reviewer focused on Prashanth's airline messaging platform.

When invoked, you MUST:
1. Read the target file before commenting (never assume its contents)
2. Check that all consumers have Dead Letter Queue configuration
3. Verify message schema validation against the company schema registry
4. Flag any hardcoded topic names — they should come from application.yaml
5. Return findings as a structured list with file:line references

If uncertain, say "I need to read X before I can assess this."
Never invent code or assume message schemas. Read first, always.
.claude/rules/kafka.md — Prompt Template
# This is a RULE (prompt context), not an agent
# It's merged into CLAUDE.md and applied to all interactions

## Kafka Rules
- All consumers must have DLQ configuration
- Use Spring Kafka's @KafkaListener — never raw KafkaConsumer
- Topic names must be read from application.yaml, never hardcoded

The 4 Key Differences

PropertyPromptAgent
Context WindowShared with youIsolated — its own
ExecutionOne shotLoop until done / budget exhausted
Tool AccessUses your session's toolsDefined in frontmatter
ParallelismSequentialCan run in parallel with other agents
💡 When to Use Which Use a prompt/rule when the instruction is a style or coding standard that should always be active. Use an agent when the task needs file exploration, multi-step reasoning, or domain expertise that you want isolated from the main conversation's context.

Personas & Model Assignment

Give your agents identity, role, and the right model

Assigning a Persona to a Prompt

A persona shapes how the AI responds — its expertise, tone, and decision-making style. In your CLAUDE.md or copilot-instructions.md:

CLAUDE.md — Persona Definition
## Your Identity
You are a senior distributed systems engineer specializing in airline operations infrastructure.
You have deep expertise in Java/Spring Boot, Kafka, Solace PubSub+, and AWS.
You think like an SRE: always consider failure modes, DLQs, and observability first.
You prefer explicit, readable code over clever one-liners.
When reviewing code, you ask: "What happens when this fails at 3am?"

Assigning a Persona to an Agent

---
name: security-auditor
description: "OWASP-focused security review for REST APIs and Spring Boot services"
tools:
  - Read
  - Bash(grep)
---

You are a security engineer with OWASP expertise reviewing airline passenger data APIs.
You are adversarial by nature — you look for what could go wrong, not what works.

Always check:
- SQL injection: are all queries parameterized?
- Auth: does every endpoint verify JWT claims?
- PII: is passenger data masked in logs?
- IDOR: can a user access another user's records by changing an ID?

Assign model per agent (SDK)
import { query } from "@anthropic-ai/claude-agent-sdk";

// Architect agent — use Opus for deep reasoning
const plan = await query({
  model: "claude-opus-4-6",
  system: "You are a distributed systems architect. Design before code.",
  prompt: "Design the Solace migration for our baggage tracking system."
});

// Code generation — use Sonnet for speed + quality balance
const code = await query({
  model: "claude-sonnet-4-6",
  system: "Implement the design. Prefer explicit over clever. Add JUnit tests.",
  prompt: plan.result
});
💡 Cost Optimization Pattern Use Opus for planning and Sonnet for execution. A well-designed plan from Opus typically reduces total Sonnet calls because the implementation is clearer — net cost is often lower than using Sonnet for everything.

Eliminating Hallucinations

Generate code that is grounded, verified, and assumption-free

🔴 What Hallucination Looks Like in Code The AI invents a method that doesn't exist. It uses a library version that has a different API. It references a DynamoDB table with the wrong attribute name. It creates an import path for a package that doesn't exist. These look like working code — until they don't.

The 7 Rules for Hallucination-Free Code

1

Read Before Answering

Add this to every CLAUDE.md: "Never speculate about code you have not opened. Read the file before referencing it." This single instruction eliminates most hallucinated method/class references.

2

Allow "I Don't Know"

Explicitly tell the agent it can say it's uncertain: "If you are not confident in a method signature or package version, say so instead of guessing." LLMs hallucinate less when you remove the pressure to always answer.

3

Ground Imports in package.json / pom.xml

Add to your config: "Only use library versions that appear in pom.xml or package.json. Never import a dependency that isn't listed."

4

Use the Spec Approval Gate (Kiro)

In Kiro, review the spec before code is generated. If the spec contains an incorrect assumption, fix it there — the code will be correct downstream.

5

Best-of-N Verification

For critical code paths, run the same prompt 2–3 times and compare outputs. Inconsistencies across runs indicate hallucinations — consistent outputs are far more likely to be grounded.

6

Cite Your Sources

Ask the agent to cite which file and line number justifies each decision: "For every code change, add a comment referencing the file you read that justified this approach."

7

Restrict External Knowledge

For domain-specific code: "Use only information from files in this repository. Do not use general knowledge about how this library typically works — read the actual implementation."

The Anti-Hallucination CLAUDE.md Block

Copy this into every CLAUDE.md
## Anti-Hallucination Rules (Non-Negotiable)

### Before Writing Code
- Read the relevant source files BEFORE writing or modifying code
- Never assume a method, field, or class exists — verify it
- Only reference imports that exist in pom.xml or package.json

### When Uncertain
- Say "I need to read X to answer this" — never guess
- If a schema or interface is not in the files you've read, ask for it
- Write a TODO comment rather than inventing an implementation

### Verification
- After writing code, trace each import back to its source file
- For any DynamoDB attribute or Kafka topic name used, confirm it in the schema docs
- Run the tests you've written — if they fail due to your code, fix it before finishing
💡 The Grounding Prompt Pattern For any complex task: start with "First, read [file list]. Then summarize what you found. Then, and only then, write the implementation." This enforces reading before writing, eliminates 90% of hallucinated references.
